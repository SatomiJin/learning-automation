Tuần 9 · Page Object Model (POM)
🏛️
Page Object Model — Kiến trúc POM
BasePage · locators tập trung · action methods · tách logic
▼
Lý thuyết — POM là gì và tại sao cần dùng?

POM (Page Object Model) là design pattern tổ chức code automation. Thay vì viết locator và action rải rác trong test, bạn tạo một class cho mỗi trang — class đó chứa locators và các method tương tác.

💡
Lợi ích POM: Khi UI thay đổi (ví dụ dev đổi id của nút), bạn chỉ cần sửa 1 chỗ trong page class — thay vì tìm và sửa trong 20 file test.

Cấu trúc thư mục chuẩn:

Python pages/base_page.py 
from playwright.sync_api import Page, expect

class BasePage:
    """Class cha chứa các method dùng chung"""

    def __init__(self, page: Page):
        self.page = page

    def navigate(self, url: str):
        self.page.goto(url)

    def wait_for_page_load(self):
        self.page.wait_for_load_state("networkidle")

    def take_screenshot(self, name: str):
        self.page.screenshot(path=f"screenshots/{name}.png")

    def get_title(self) -> str:
        return title()
Python pages/login_page.py 
from playwright.sync_api import expect
from pages.base_page import BasePage

class LoginPage(BasePage):
    # Locators — tập trung tại đây
    URL          = "https://www.saucedemo.com"
    USERNAME_INP = "#user-name"
    PASSWORD_INP = "#password"
    LOGIN_BTN    = "#login-button"
    ERROR_MSG    = "[data-test='error']"

    def open(self):
        self.navigate(self.URL)
        return self

    def login(self, username: str, password: str):
        self.page.fill(self.USERNAME_INP, username)
        self.page.fill(self.PASSWORD_INP, password)
        self.page.click(self.LOGIN_BTN)
        return self  # cho phép method chaining

    def get_error_message(self) -> str:
        return self.ERROR_MSG).inner_text()

    def assert_error_visible(self):
        expect(self.page.locator(self.ERROR_MSG)).to_be_visible()
        return self

Python tests/conftest.py 
import json, os
import pytest
from dotenv import load_dotenv

load_dotenv()   # đọc file .env

## Fixture đọc test users từ JSON
@pytest.fixture(scope="session")
def test_users():
    with open("test_data/users.json") as f:
        return json.load(f)

## Fixture cấu hình base URL từ môi trường
@pytest.fixture(scope="session")
def base_url():
    return os.getenv("BASE_URL", "https://www.saucedemo.com")

## Factory function — tạo test data động
@pytest.fixture
def make_user():
    def _make(role="standard", **overrides):
        defaults = {
            "username": f"test_user_{role}",
            "email":    f"{role}@test.com",
            "role":     role
        }
        return {**defaults, **overrides}   # overrides thắng defaults
    return _make

## Dùng trong test
def test_admin_panel(make_user, page, base_url):
    admin = make_user(role="admin", username="actual_admin")
    page.goto(base_url)
    page.fill("#username", admin["username"])

@allure.feature("Authentication")
@allure.story("Login")
def test_login_success(page):
    with allure.step("Mở trang login"):
        page.goto("https://www.saucedemo.com")

    with allure.step("Điền credentials"):
        page.fill("#user-name", "standard_user")
        page.fill("#password", "secret_sauce")

    with allure.step("Click Login"):
        page.click("#login-button")
        screenshot = page.screenshot()
        allure.attach(screenshot, name="After login",
                      attachment_type=allure.attachment_type.PNG)

Debugging & Flaky Test Strategies
Trace viewer · headed mode · retries · isolating failures
▼
Lý thuyết — Flaky test là gì?

Flaky test = test lúc pass lúc fail không có lý do rõ ràng. Đây là vấn đề phổ biến nhất trong automation. Nguyên nhân chính: timing issue, data không sạch, phụ thuộc giữa tests, animation chưa xong.

Bash Debug commands 
# Chạy headed để xem browser
pytest tests/ --headed

# Chạy chậm lại 1000ms mỗi action
pytest tests/ --headed --slowmo 1000

# Bật trace để debug sau khi fail
pytest tests/ --tracing=on

# Xem trace file
playwright show-trace traces/test_login.zip

# Retry tự động khi fail
pytest tests/ --retries=2

Python conftest.py — auto trace on fail 
@pytest.fixture(autouse=True)
def trace_on_failure(page, request):
    page.context.tracing.start(screenshots=True, snapshots=True)
    yield
    # Chỉ lưu trace khi test FAIL
    if hasattr(request.node, "rep_call") and request.node.rep_call.failed:
        path = f"traces/{request.node.name}.zip"
        page.context.tracing.stop(path=path)
        print(f"\nTrace saved: {path}")
    else:
        page.context.tracing.stop()
Triệu chứng Nguyên nhân thường gặp Cách fix
Element not found Page chưa load xong Dùng wait_for / expect().to_be_visible()
Click không hoạt động Element bị che/đang animate Chờ animation xong, scroll into view
Data sai Test trước để lại dữ liệu Cleanup trong fixture teardown
Timeout Network chậm ở CI Tăng global timeout trong playwright.config
❌
Đừng chỉ tăng retry để che flaky test. Retry là lưới an toàn tạm thời — hãy tìm root cause thật sự. Trace viewer thường chỉ ra vấn đề trong 5 phút.

▼
Lý thuyết
YAML .github/workflows/playwright.yml 
name: Playwright Tests

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          playwright install --with-deps chromium

      - name: Run Playwright tests
        run: pytest tests/ -v --alluredir=allure-results

      - name: Upload Allure results
        if: always()   # upload kể cả khi test fail
        uses: actions/upload-artifact@v3
        with:
          name: allure-results
          path: allure-results/

      - name: Upload screenshots on failure
        if: failure()
        uses: actions/upload-artifact@v3
        with:
          name: screenshots
          path: screenshots/

Tuần 12 · Best Practices & Code Quality
🏆
Best Practices — Viết test như senior
Test isolation · naming · no sleep · project structure chuẩn
▼
Những điều KHÔNG BAO GIỜ làm
❌
Danh sách đen:
• time.sleep() — dùng expect().to_be_visible() thay thế
• Locator fragile như //div[3]/span[1]
• Test phụ thuộc nhau (test B phải chạy sau test A)
• Credentials hardcode trong test file — dùng .env
• Bắt except Exception: pass — che giấu lỗi thật
✅
Checklist test chất lượng:
• Mỗi test độc lập, chạy được riêng lẻ
• Tên test rõ ràng: test_login_with_invalid_password_shows_error
• Dùng @pytest.fixture cho setup/teardown
• Một test chỉ kiểm tra một điều (Single responsibility)
• Comment giải thích tại sao chứ không phải làm gì
